# Roomly Unified Coursework

Единый курсовой проект по дисциплине «Создание программного обеспечения»: информационная система бронирования отелей Roomly и связанный модуль управления персоналом Roomly Staff.

## Архитектура

Проект оформлен как распределенный монолит:

- `main/` — основной frontend Roomly и backend REST API для отелей, бронирований, поддержки, пользователей и аудита;
- `staff/` — frontend панели персонала и backend REST API для сотрудников, смен, задач, отсутствий, отчетов и аудита;
- связь модулей выполнена через REST-события: при создании, изменении, отмене или удалении бронирования основной backend отправляет событие в Staff API, а Staff API создает или обновляет задачу заселения.

## Запуск

```bash
npm install
npm run install:all
npm run dev:all
```

После запуска:

- основной сайт: http://127.0.0.1:5173
- основной API: http://127.0.0.1:3000/api
- Staff-панель: http://127.0.0.1:5174
- Staff API: http://127.0.0.1:3100/api

- `ivan.petrov@roomly.local` / `Admin12345` — сотрудник, видит раздел «Персонал»

Roomly Staff:

- `admin@roomly.local` / `Admin12345` — полный доступ
- `ivan.petrov@roomly.local` / `Admin12345` — сотрудник, без изменения смен и выходных

## Проверка связи модулей

1. Запустить оба backend и frontend через `npm run dev:all`.
2. Открыть основной сайт, войти как `user@roomly.local` и создать бронирование.
3. Открыть Staff-панель, войти как `admin@roomly.local`.
4. Перейти в раздел «Задачи»: должна появиться задача «Подготовить заселение гостя ...».
5. Перейти в раздел «Аудит»: появится событие `integration.booking.created`.

## Что исправлено при объединении

- Синхронизированы ID отелей между модулями: `ikeja-lagos`, `jabi-dubai`, `arara-lagos`, `lekki-paris`, `ikeja-london`.
- Добавлен защищенный endpoint Staff API: `POST /api/integration/booking-event`.
- Основной backend отправляет события бронирований в Staff API через `STAFF_API_URL` и `ROOMLY_INTEGRATION_TOKEN`.
- В Staff backend усилены проверки связей: сотрудник, задача, смена и отдел должны относиться к одному отелю.
- Улучшена устойчивость JSON-хранилища Staff: операции обновления поставлены в очередь, запись идет через временный файл, создаются резервные копии.
- Исправлена проверка токена Staff: добавлены проверки длины подписи и безопасная обработка некорректных токенов.
- Интерфейс Staff фильтрует отделы и сотрудников по выбранному отелю и экранирует пользовательские значения при выводе в таблицы.

## Что добавлено в финальной версии

- Раздел **Бронирования** в Staff-панели: сотрудники и администратор видят заявки с основного сайта и меняют их статус.
- Раздел **Обращения** работает как диалог: клиент пишет вопрос, сотрудник отвечает, администратор закрывает обращение.
- В личном кабинете пользователя отображаются **мои бронирования** и **мои обращения** с ответами поддержки.
- Добавлены фильтры по статусам для задач, бронирований и обращений.
- Добавлены карточки статистики: новые бронирования, открытые обращения, задачи, смены, сотрудники.
- Добавлена страница **Доступ запрещён** для ролей без прав.
- В Staff-панели отображаются журналы действий Staff-модуля и основного сайта.
- Подготовлены документы `docs/ROLE_MATRIX.md`, `docs/DEMO_SCENARIO.md`, `docs/IMPROVEMENTS.md`.

## Роли

- **Гость** — просматривает основной сайт и может создать обращение в поддержку, но не может бронировать.
- **Пользователь** — может бронировать отели, смотреть свои бронирования и вести диалог по обращениям.
- **Сотрудник** — входит в Staff-панель, работает с задачами, обращениями и бронированиями, но не управляет сменами и выходными.
- **Администратор** — имеет полный доступ, включая смены, сотрудников, закрытие обращений и аудит.

Подробная таблица прав находится в `docs/ROLE_MATRIX.md`.
